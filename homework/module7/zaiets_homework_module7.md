## 1. Запуск контейнера (недоступного з хоста)
Запускаємо веб-сервер Nginx у фоновому режимі без "прокидання" портів:
`docker run -d --name my-web nginx`

## 2. Діагностика проблеми
**Перевірка портів усередині контейнера:**
`docker exec my-web ss -tulpn`
*(Результат показує, що процес Nginx успішно слухає порт 80 всередині контейнера).*

**Перевірка port mapping (прокидання портів) на хості:**
`docker port my-web`
*(Результат порожній. Жодних портів не прокинуто).*

**Де саме проблема:**
Застосунок працює та слухає порт 80, але цей порт існує виключно в ізольованій мережі самого контейнера (bridge network). Оскільки ми не вказали прапорець `-p` при запуску, Docker не створив правило маршрутизації (NAT), яке б направляло трафік з нашого основного комп'ютера (хоста) всередину контейнера.

## 3. Виправлення конфігурації
Зупиняємо та видаляємо "зламаний" контейнер:
`docker stop my-web && docker rm my-web`

Запускаємо контейнер правильно, зв'язуючи порт 8080 на хості з портом 80 у контейнері:
`docker run -d --name my-web -p 8080:80 nginx`

**Перевірка доступності:**
`curl http://localhost:8080`
*(Отримуємо у відповідь HTML-код сторінки Nginx — сервіс успішно доступний).*

## Висновок: чому IP контейнера ≠ доступність сервісу
Кожен Docker-контейнер отримує свою власну внутрішню IP-адресу (наприклад, 172.17.0.2). Проте ця адреса доступна лише зсередини Docker-мережі. Навіть якщо застосунок ідеально працює на цьому внутрішньому IP, для зовнішнього світу (і для браузера на хості) він не існує. Доступність сервісу визначається не наявністю внутрішнього IP, а налаштованим механізмом Port Mapping (`-p host_port:container_port`), який дозволяє Docker-демону перехоплювати запити на певний порт хоста і перенаправляти їх на внутрішній IP та порт контейнера.
